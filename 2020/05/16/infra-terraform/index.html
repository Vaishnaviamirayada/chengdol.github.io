Terraform Quick Start

Posted on 2020-05-16 | Edited on 2020-05-22 | In Infra
Symbols count in article: 14k

Terraform

From Hashicorp, removing manual build process, adopting declarative approach to deploy infrastructure as code, reusable, idempotent and consistent repeatable deployment.

Terrafrom provision GKS
https://learn.hashicorp.com/terraform/kubernetes/provision-gke-cluster

This post from IBM can give you a good overview:
https://www.ibm.com/cloud/learn/terraform
Terraform vs Kubernetes
Terraform vs Ansible

CSDN:
https://www.cnblogs.com/sparkdev/p/10052310.html

Terraform 文档非常informative，结构清晰:
https://www.terraform.io/docs/

Terraform之于cloud infra就相当于 helm之于K8s, 大大简化了操作复杂性，自动快速部署，同时做到了复用，versioning等特性。但首先得去了解cloud provider中提供的资源的用途，搭配。

Play around

resource files in github:
https://github.com/ned1313/Getting-Started-Terraform

what are needed:
Aws account, may get charged, ask for GCP free access.

Composition

Terraform executable: download from web
Terraform files: using hashicorp configure language DSL
Terraform plugin: interact with provider: AWS, GCP, Azure, etc
Terraform state file: json and don’t touch it

You can have multiple terraform file: .tf, when run terraform it will stitch them together to form a single configuration. 比如把variables, outputs, resources, tfvars分开。

tfvars file be default named as terraform.tfvars. otherwise when run plan you need to specify the file path.

Command

If you update the terraform file with different configuration, rerun init, plan and apply.

Provisioning k8s in GCP:
https://learn.hashicorp.com/terraform/kubernetes/provision-gke-cluster

Syntax

Hashicorp configuration language:
https://www.terraform.io/docs/configuration/index.html

参考这些例子中的tf文件，格式还是比较固定的
https://github.com/ned1313/Getting-Started-Terraform

怎么知道resource的名称呢? 有很多provider现成的resource可用:
https://www.terraform.io/docs/configuration/resources.html
还有random provider，比如产生随机数

Object types:

  • string
  • number (integer/decimal)
  • bool (true/false)
  • list (index start from 0)
  • map

Provisioner

类似于ansible的配置操作，在deploy infrastructure 之后。
post-deployment configuration, last resort, prefer ansible/puppet or chef.
provisioner can be local or remote

common provisioner:

  • file
  • local-exec
  • remote-exec

Functions

https://www.terraform.io/docs/configuration/functions.html

You can experiment with functions in Terraform console, this can help with troubleshooting.

Provider

Support mutliple providers, all written in Go.
https://www.terraform.io/docs/providers/index.html

Resource arguments

https://www.terraform.io/docs/configuration/resources.html#depends_on-explicit-resource-dependencies
common ones:

  • depends_on: make sure terraform creates things in right order
  • count: create similar resources
  • for_each: create resources not similar
  • provider

Variables

Other way to use variables rather than specifying in single .tf file.

The scenario, we need development, QA/UAT, production environment, how to implement via one configuratio and multiple inputs?

The variable can be from:

  • file
  • environment variable
  • var option

You can override variables and precedence, select value based on environment, for example:

Workspace

Workspace is the recommended way to working with multiple environments, for example:

  • state management
  • variables data
  • credentials management

State file example, we have dev, QA, prod three environments, put them each into separate folder, when run command, specify the input and output:

Workspace example, there is a terraform.workspace built-in variable can be used to indicate the workspace currently in, then use it in map variable to select right value for different environment. (不用再去分别创建不同的folder for different environment了)

Managing secrets

Hashicorp vault is for this purpose. it can hand over credentials from cloud provider to terraform and set ttl for the secrets.

Or you can use environment variable to specify the credentials, terraform will pick it automatically, but bear in mind to use the right env var name.

Module

Make code reuse eaiser:
https://www.terraform.io/docs/configuration/modules.html

Terraform registry, similar concept with Helm, Docker:
https://registry.terraform.io/
Using module block to invoke local or remote modules.

  1. root module
  2. support versioning
  3. provider inheritance

Module components:

  • variables input
  • resources
  • output values (calling part will take this in)
